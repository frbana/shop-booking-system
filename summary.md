# AI辅助编程实训总结：Next.js + Flask 店铺预约系统

本次实训我完成的是一个 Next.js + Flask 的店铺预约系统，功能从最初的预约时段展示、提交预约、活动列表、个人预约查询和取消，逐步扩展到用户注册登录、用户资料维护、头像上传、示例数据初始化、API 文档、README 文档和线上部署说明。整个过程不是简单把代码写出来，而是把一个前后端分离项目从本地跑通、接口联调、数据持久化、公开仓库展示到线上部署都走了一遍。

## 一、开发踩坑难点与解决步骤

第一个比较明显的坑是 Mac 本地环境配置。最开始安装 Node.js、Python 等基础环境时，Homebrew 会出现网络超时或下载很慢的问题，导致依赖迟迟装不上。我的解决步骤是先确认本机是否已经有 `node -v`、`npm -v`、`python3 --version`，能用就不重复安装；如果必须用 Homebrew，就先执行 `brew update`，网络不稳定时换时间段重试。后端依赖没有强制依赖全局环境，而是在 `backend` 目录用 `python3 -m venv .venv` 创建虚拟环境，再通过 `pip install Flask Flask-SQLAlchemy flask-cors python-dotenv gunicorn` 安装需要的包，避免把系统 Python 环境弄乱。

第二个坑是前后端跨域和端口问题。Mac 上 `5000` 端口被系统的 `ControlCe` 占用，直接访问 `localhost:5000` 还出现过拒绝访问，所以我改用 `FLASK_PORT=5001 python app.py` 启动 Flask 后端。同时，前端 Next.js 默认跑在 `localhost:3000`，如果后端没有配置 CORS，浏览器就会报 `Failed to fetch`。最后的解决方式是在后端启动时显式设置 `CORS_ORIGINS=http://localhost:3000,http://127.0.0.1:3000`，前端通过 `frontend/.env.local` 配置 `NEXT_PUBLIC_API_BASE_URL=http://127.0.0.1:5001`，并且修改环境变量后必须重启 `npm run dev`，这样接口请求才会指向正确后端。

第三个难点是预约时段库存校验。预约系统不能只在前端判断“剩余人数大于 0”，因为多个用户同时提交时可能产生超卖。后端接口 `/api/book-order` 里先校验姓名、手机号和 `slot_id`，再判断时段是否存在，并限制同一手机号在同一时段只能有一笔正常预约。真正扣库存时，不是简单读取后再加一，而是使用 SQLAlchemy 的条件更新：只有 `booked_count < max_capacity` 时才把 `booked_count` 加 1。如果更新行数不是 1，就说明该时段已经约满，需要回滚事务并返回“该时段已约满”。这个处理让我理解到库存逻辑必须放在后端和数据库层面兜底。

第四个坑是线上部署报错。前端部署到 Vercel、后端部署到 Zeabur 时，最容易错的是路径、启动命令和环境变量。前端仓库根目录不是 Next.js 项目根目录，所以 Vercel 的 Root Directory 必须设置为 `frontend`，环境变量要配置 `NEXT_PUBLIC_API_BASE_URL=https://你的-zeabur-后端域名.zeabur.app`。后端在 Zeabur 里要把 Root Directory 设置为 `backend`，并配置 `ZBPACK_PYTHON_ENTRY=app.py`、`CORS_ORIGINS`、`SECRET_KEY` 等环境变量。如果部署后前端接口失败，就先打开 Zeabur 的 `/api/health` 看后端是否活着，再检查 Vercel 请求地址是否仍然指向 localhost。

## 二、AI辅助编程心得

这次实训里我最明显的体会是，AI 能提高效率，但前提是 Prompt 要具体。比如我让 AI 生成 README、API 文档、数据库表、后端接口时，如果只说“帮我写文档”，得到的内容会比较泛；当我明确要求“包含项目简介、4 大页面、4 个后端接口、Mac 安装命令、Vercel/Zeabur 部署、目录结构、交付物清单”后，AI 输出就更接近实训要求。后续新增“用户注册登录、手机号唯一绑定、头像保存到数据库”时，我也需要把业务规则说清楚，否则 AI 可能只做前端本地保存，而不是落到 SQLite 数据库。

AI 在生成数据表、接口和页面方面确实节省了大量时间。数据库从 `time_slot`、`booking`、`activity` 扩展到 `user` 表，后端从基础的查询、预约、取消接口扩展到注册、登录、保存用户资料接口，前端从 4 个页面扩展到更完整的个人中心，这些工作如果完全手写会比较耗时。AI 可以先生成一版可运行代码，我再根据项目实际情况修正端口、跨域、字段、样式和交互。

另一个收获是 Code Review 的价值。AI 不是只负责“生成代码”，也可以用来做代码评审。比如之前对接口返回格式、错误处理、手机号校验、数据库事务、前端请求封装、生产环境 API 地址等问题做过检查，然后再根据评审结果修复。这个过程让我意识到，AI 生成的代码不能直接当最终答案，仍然要通过构建、接口测试和逻辑检查来确认。尤其是登录注册这种涉及密码的功能，需要使用哈希保存，而不是明文保存，这就是 Code Review 能帮忙发现和强化的地方。

## 三、工程化实训收获

这次项目让我对工程化交付有了更实际的认识。以前写小项目时，我更多关注“页面能不能打开”，但这次需要考虑 Git 多日期规范提交、公开 GitHub 仓库、线上部署、README、API 文档、Prompt 归档、代码评审记录、示例数据脚本等内容。每一个文件都不是形式主义，而是让别人可以复现、检查和使用这个项目。

Git 多日期规范提交的意义在于保留开发过程，不是最后一次性提交所有文件。Day1 可以体现数据库、模型、基础接口和页面骨架；Day2 可以体现代码评审、问题修复和部署教程；Day3 可以体现 README、API 文档、用户功能和总结文档。这样的提交记录能让老师或评审者看到项目是逐步完成的，也方便自己回溯问题。

公开仓库和线上部署也很重要。本地运行只能证明项目在我电脑上可用，而部署到 Vercel 和 Zeabur 后，别人通过 URL 就能访问，这才更接近真实项目交付。部署过程中暴露出的 CORS、环境变量、启动命令、SQLite 持久化等问题，也都是本地开发不一定能遇到的工程问题。

最后，全套文档留痕让我明白实训考核不只是看代码行数。README 让别人快速了解项目和启动步骤，`api_doc.md` 让前后端接口有据可查，`prompt_log.md` 记录 AI 辅助过程，`summary.md` 总结踩坑和收获，`docs/code_review` 保留评审过程。这些内容共同证明项目不是临时拼出来的，而是经过设计、开发、调试、优化和部署的完整成果。对我来说，这次实训最大的收获不是学会某一个语法点，而是知道了一个可展示、可部署、可交付的全栈项目应该包含哪些必要环节。
